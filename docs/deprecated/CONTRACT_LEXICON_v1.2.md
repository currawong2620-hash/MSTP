## -1. ARCHITECTURE TYPE GOVERNANCE

**Status:** NORMATIVE  
**Scope:** All architecture data contracts  
**Change policy:** versioned only  
**Authority:** SSP-referenced document

---

### -1.1. Single Source of Truth

Настоящий документ является **ЕДИНСТВЕННЫМ источником истины**
о допустимых архитектурных типах данных и их полях.

Любой тип, поле или enum-значение,
отсутствующие в данном документе,
считаются **ЗАПРЕЩЁННЫМИ**.

---

### -1.2. Ownership of Types

Архитектурные типы данных:

- **НЕ определяются** в исполняемом коде
- **НЕ определяются** в технических заданиях
- **НЕ принадлежат** отдельным модулям или ролям

Архитектурные типы принадлежат
**архитектуре системы в целом**, а не реализации.

---

### -1.3. Rule for Master-Chat

Ни один Master-Chat **НЕ ИМЕЕТ ПРАВА**:

- вводить новый архитектурный тип,
- вводить новое поле существующего типа,
- вводить новый enum или enum-значение,
- переименовывать существующие элементы,

если соответствующее изменение
**НЕ зафиксировано в новой версии данного документа**.

При отсутствии подходящего типа
**работа должна быть остановлена**.

---

### -1.4. Separation from Implementation

Код **МОЖЕТ ТОЛЬКО ИСПОЛЬЗОВАТЬ**
архитектурные типы, определённые в данном документе.

Код **НЕ ИМЕЕТ ПРАВА**:

- определять структуру типов,
- расширять поля,
- интерпретировать типы «по смыслу».

---

### -1.5. Audit Enforcement

Любой Audit-Chat обязан:

- отклонить код,
- отклонить техническое задание,
- отклонить обсуждение,

если используется тип, поле или enum,
отсутствующие в настоящем документе.

---

### -1.6. Interpretative Prohibition

Если возникает сомнение,
соответствует ли существующий тип
требуемой сущности,

интерпретация считается **НЕДОПУСТИМОЙ**.

В этом случае:
1. код не пишется,
2. ТЗ не формируется,
3. сначала обновляется Lexicon.

---

**End of Architecture Type Governance**

- Имена типов и полей **буквально обязательны**

## 3. Snapshot

### 3.1. Назначение
Снимок наблюдаемого состояния мира в один момент времени.

### 3.2. Структура

```

Snapshot
├── market
├── position
├── time
└── constraints

```

---

### 3.3. Snapshot.market

**Тип:** MarketSnapshot

| Поле | Семантический тип | Примечание |
|----|----|----|
| symbol | symbol | торговый инструмент |
| tf | timeframe | таймфрейм |
| open | price | цена открытия бара |
| high | price | максимум бара |
| low | price | минимум бара |
| close | price | цена закрытия бара |
| volume | volume | объём бара |
| bid | price | текущий bid |
| ask | price | текущий ask |
| spread | price | ask - bid |
| last_closes | price[3] | закрытия трёх последних ЗАКРЫТЫХ баров |

#### 3.3.X. Semantics of `last_closes` (НОРМАТИВНО)

* `last_closes` содержит **ТОЛЬКО закрытые бары**
* `last_closes[0]` — закрытие **последнего закрытого бара**
* `last_closes[1]` — закрытие предыдущего бара
* `last_closes[2]` — закрытие ещё одного предыдущего бара
* массив имеет **фиксированную длину = 3**
* порядок элементов **строго обязателен**
* значения формируются **Observer**
* другие роли **НЕ имеют права модифицировать** массив

* `last_closes`:
  * НЕ является индикатором
  * НЕ содержит вычисленных значений
  * НЕ агрегируется
  * НЕ интерпретируется как тренд

* Наличие `last_closes` НЕ разрешает:
  * хранение состояния в DecisionMaker
  * вычисление индикаторов
  * расширение окна истории без новой версии Lexicon
* `DecisionMaker` и другие роли
  МОГУТ ТОЛЬКО читать `last_closes`

## 4. Intent

### 4.1. Назначение
Выражение желания действия без гарантии исполнения.

### 4.2. Структура

Intent
├── type
├── direction
└── confidence

### 4.3. Поля

| Поле | Тип | Примечание |
|----|----|----|
| type | intent_type | категория намерения |
| direction | direction | +1 / -1 / 0 |

### 4.4. intent_type (enum)

- `NO_ACTION`
- `WANT_OPEN`
- `WANT_CLOSE`
- `WANT_HOLD`

---

## 5. PolicyAdjustedIntent

### 5.1. Назначение
Намерение после применения политики владения позицией.

### 5.2. Структура

```

PolicyAdjustedIntent
├── type
├── direction
├── volume
└── tag

```

### 5.3. Поля

| Поле | Тип | Примечание |
|----|----|----|
| type | intent_type | категория |
| direction | direction | направление |
| volume | volume | допустимый объём |
| tag | string | идентификатор политики |

---
### 5.4. Семантические правила для `PolicyAdjustedIntent` (НОРМАТИВНО)

#### 5.4.1 Общие правила

* `PolicyAdjustedIntent` является **политически допустимым намерением**,
  сформированным на основе `Intent` и фактов владения позицией.
* `PolicyAdjustedIntent` **НЕ является** командой исполнения.

---

#### 5.4.2 Правила для `type = WANT_OPEN`

* `direction` **ОБЯЗАТЕЛЕН**
* `direction ∈ { +1, -1 }`
* `volume` **ОБЯЗАТЕЛЕН**

---

#### 5.4.3 Правила для `type = WANT_CLOSE`

* `direction` **ДОЛЖЕН БЫТЬ = 0**
* `volume` **ОБЯЗАТЕЛЕН**
* `direction` **НЕ интерпретируется** как рыночное направление
* `WANT_CLOSE` обозначает **факт запроса закрытия позиции**,  
  а не направление торгового действия

---

#### 5.4.4 Архитектурное следствие

* Для `WANT_CLOSE` **направление позиции выводится из факта владения**,  
  а не из `direction`
* Это устраняет неоднозначность при передаче
  `PolicyAdjustedIntent → Decision(action = CLOSE)

---

## 6. Decision

### 6.1. Назначение

`Decision` фиксирует **разрешённую к исполнению команду**
или **запрет на исполнение**, полученные в результате арбитража допустимости.

`Decision`:

* является **ЕДИНСТВЕННЫМ входом Executor**
* содержит **всю минимально достаточную информацию**
  для исполнения **без чтения других архитектурных типов**
* **НЕ содержит** стратегической логики
* **НЕ содержит** рыночной интерпретации
* **НЕ объясняет**, *почему* решение принято

---

### 6.2. Структура

```
Decision
├── status
├── action
├── direction
├── volume
├── symbol
└── reason
```

---

### 6.3. Поля

| Поле      | Тип             | Примечание                                                  |
| --------- | --------------- | ----------------------------------------------------------- |
| status    | decision_status | итог арбитража                                              |
| action    | decision_action | тип разрешённого действия                                   |
| direction | direction       | направление действия                                        |
| volume    | volume          | разрешённый объём                                           |
| symbol    | symbol          | торговый инструмент                                         |
| reason    | string          | код причины (для логирования, не интерпретируется Executor) |

---

### 6.4. decision_status (enum)

| Значение | Значение                                           |
| -------- | -------------------------------------------------- |
| ACCEPT   | действие разрешено                                 |
| REJECT   | действие запрещено                                 |
| MODIFY   | действие разрешено с модифицированными параметрами |

---

### 6.5. decision_action (enum)

| Значение | Значение        |
| -------- | --------------- |
| OPEN     | открыть позицию |
| CLOSE    | закрыть позицию |

---

### 6.6. Семантические правила (НОРМАТИВНО)

#### 6.6.1 Общие правила

* `Decision` **НЕ является** намерением
* `Decision` **НЕ является** результатом исполнения
* `Decision` является **разрешённой командой**

---

#### 6.6.2 Правила для `status = REJECT`

* `action`, `direction`, `volume`, `symbol`
  **НЕ ДОЛЖНЫ интерпретироваться**
* `Executor` **НЕ вызывается**
* `Decision` используется **только для фиксации факта запрета**

---

#### 6.6.3 Правила для `status = ACCEPT | MODIFY`

* `action` **ОБЯЗАТЕЛЕН**
* `symbol` **ОБЯЗАТЕЛЕН**
* `volume` **ОБЯЗАТЕЛЕН**
* `direction`:

  * **ОБЯЗАТЕЛЕН** при `action = OPEN`
    и обозначает направление открытия позиции

  * **ОБЯЗАТЕЛЕН** при `action = CLOSE`
    и обозначает направление позиции, подлежащей закрытию

`Executor` использует **ТОЛЬКО эти поля**
и **НЕ делает выводов** о происхождении параметров.

---

#### 6.6.4 Различие ACCEPT и MODIFY

* `ACCEPT` — параметры допустимы без изменений
* `MODIFY` — параметры были изменены Risk / Arbiter

Для `Executor` **различия НЕТ**.
Он использует итоговые значения полей.

---

### 6.7. Архитектурные запреты

Запрещено:

* добавлять другие поля без новой версии Lexicon
* трактовать `Decision` как стратегию
* трактовать `Decision` как рыночный прогноз
* читать `Decision.reason` для логики исполнения

---

### 6.8. Архитектурное следствие (критическое)

`Decision` содержит **всю информацию**, необходимую Executor,
чтобы выполнить MT5-действие:

* без чтения `Snapshot`
* без чтения `Intent`
* без чтения `PolicyAdjustedIntent`
* без интерпретации рынка
* без догадок
